# SDU Archive 项目审查报告

**审查日期**: 2026-02-20
**项目**: 山东大学校史互动档案库（RAG + 题库）
**审查视角**: 产品经理 + 全栈工程师

---

## 一、产品经理视角

### 1.1 产品定位与价值主张

**优势**:
- ✅ **清晰的用户价值**: 结合知识检索（RAG）与互动题库的双引擎设计，既满足信息获取需求，又提供学习互动体验
- ✅ **低门槛部署**: Docker Compose一键启动，适合教育机构快速落地
- ✅ **差异化设计**: "暂无记载"机制降低AI幻觉，体现对历史档案真实性的重视

**问题**:
- ⚠️ **产品定位模糊**: 同时服务"档案查询"和"互动学习"两个场景，但用户旅程割裂
  - 档案查询用户可能不关心答题积分
  - 答题用户可能不需要深度RAG检索
  - 建议：明确核心场景优先级，或通过用户画像分层设计

### 1.2 用户体验问题

#### 1.2.1 关键流程缺陷

**注册/登录流程**:
- ❌ **密码复杂度无要求**: 允许弱密码（如"123"），存在安全隐患
- ❌ **无邮箱验证**: 可随意注册不存在的邮箱，缺乏真实性验证
- ⚠️ **默认管理员账号风险**: README 明文展示 `admin@example.com / admin123`，生产环境极易被攻击

**首页登录体验**:
- ⚠️ **强制登录才能检索**: 合理但未说明原因，可能劝退潜在用户
  - 建议：提供3-5次免费体验额度，或展示示例问答激发兴趣

**RAG检索体验**:
- ❌ **无加载状态反馈**: 查询时仅显示"检索中..."，无进度指示，长时间等待焦虑
- ❌ **引用来源不可点击**: `citations` 仅展示 `source_url` 文本，无法跳转查看原文
- ⚠️ **检索结果质量差**:
  - 当前实现仅返回前 `top_k` 个 chunk 的拼接文本，无 LLM 生成总结
  - 用户得到的是"原文片段堆砌"而非"回答"，体验类似关键词搜索
  - 建议：集成 LLM（如 OpenAI Chat API）对检索结果进行归纳总结

**题库答题体验**:
- ❌ **题目选择低效**: 下拉菜单展示全量题目，题目多时体验差
  - 建议：分页或随机推荐未答题目
- ❌ **无题目详情页**: 无法查看题目解析（虽后端支持 `explanation` 字段）
- ❌ **无答题历史**: 用户无法回顾答错的题目，学习闭环不完整
- ⚠️ **防重复机制过严**: 答错题目无法重新作答，不符合学习场景

#### 1.2.2 管理功能问题

**用户管理**:
- ✅ **功能完整**: 支持列表、过滤、角色变更、状态切换、密码重置
- ⚠️ **批量操作缺失**: 无法批量禁用/删除用户，运营效率低
- ❌ **无操作日志**: 管理员变更用户角色/密码无审计记录，存在合规风险

**内容管理（Chunk编辑）**:
- ✅ **切片级编辑**: 支持搜索、编辑、删除并重新嵌入，精细化管理
- ⚠️ **批量导入缺失**: 历史档案通常成百上千，仅支持单文件上传效率低
- ❌ **无版本控制**: 编辑 chunk 后无历史版本，误操作无法回滚

**题库管理**:
- ✅ **CSV批量导入**: 解决大规模题目录入问题
- ❌ **无题目预览**: 创建/编辑题目时无法预览最终展示效果
- ❌ **无题目分类**: 所有题目混在一起，无法按主题/难度分类

### 1.3 功能缺失

#### 核心功能缺失
1. **无用户反馈机制**: RAG回答无"有用/无用"评价，无法迭代优化
2. **无内容审核流程**: 用户上传档案直接入库，无管理员审核环节（适用于开放注册场景）
3. **无搜索历史**: 用户无法查看自己的检索记录，重复输入体验差
4. **无通知系统**: 管理员操作（重置密码、禁用账号）无邮件/站内信通知

#### 数据分析不足
- Dashboard 仅展示聚合指标（总用户数、答题数），缺乏:
  - 用户活跃趋势（DAU/MAU）
  - 检索热词统计
  - 题目难度分析（平均用时、首次正确率）
  - 文档上传趋势

### 1.4 移动端体验

**优势**:
- ✅ 采用 Tailwind CSS 响应式设计，基础适配良好

**问题**:
- ⚠️ **长文本阅读差**: RAG 返回的长文本在小屏幕上难以浏览（无折叠、无摘要）
- ❌ **图表无响应式**: Dashboard 的 Recharts 在手机端显示拥挤，X轴文字重叠
- ❌ **无 PWA 支持**: 无法添加到桌面、无离线缓存

### 1.5 商业化/增长潜力

**当前限制**:
- 无付费/增值功能设计（如高级检索、专属题库）
- 无社交传播机制（如答题PK、分享排行榜）
- 无激励体系深化（积分无兑换/特权机制）

## 二、全栈工程师视角

### 2.1 架构设计

**优势**:
- ✅ **技术栈现代化**: FastAPI + React + PostgreSQL + MinIO，主流且易维护
- ✅ **容器化部署**: Docker Compose 一键启动，环境一致性好
- ✅ **pgvector 向量检索**: 原生集成，避免引入额外向量数据库，降低运维成本

**问题**:
- ❌ **单体架构瓶颈**: API、前端、数据库、对象存储紧耦合在一个 docker-compose 中
  - 无法独立扩缩容（如答题高峰期仅扩展 API 层）
  - 单点故障风险（Postgres 挂了整个服务不可用）
- ⚠️ **缺乏缓存层**:
  - Dashboard KPI 每次实时查询数据库，高并发下性能差
  - RAG 检索无热点查询缓存（Redis）
- ❌ **无异步任务队列**:
  - 文档上传后立即同步分块+嵌入（100MB文件会阻塞请求）
  - 建议：引入 Celery/RQ 处理耗时任务

### 2.3 部署与运维

**优势**:
- ✅ `scripts/smoke.sh` 提供基础冒烟测试
- ✅ 环境变量配置灵活

**问题**:
- ❌ **无健康检查**: Docker Compose 未配置 `healthcheck`，容器假死无法自动重启
- ❌ **无日志收集**: 日志仅输出到 stdout，无持久化存储（生产环境无法排查历史问题）
- ❌ **无监控告警**: 无 Prometheus/Grafana 监控 API 响应时间、数据库连接数
- ⚠️ **备份策略缺失**: PostgreSQL 数据无自动备份脚本
- ❌ **HTTPS 未配置**: 生产环境需配置 SSL 证书（Let's Encrypt）

**安全加固**:
- ❌ **MinIO 凭证硬编码**: `docker-compose.yml` 明文存储 `MINIO_ROOT_USER/PASSWORD`
- ⚠️ **数据库端口暴露**: `15433` 映射到宿主机，可被扫描攻击
  - 建议：生产环境仅内网访问，通过堡垒机管理
- ❌ **无 WAF**: API 无 Web 应用防火墙，易遭 SQL 注入、XSS 攻击

## 三、优先级修复建议

### P0（高危，立即修复）
1. **CORS 配置过宽**: `allow_origins=["*"]` → 具体域名白名单
2. **默认管理员密码泄露**: README 移除明文密码，强制首次登录修改
3. **MinIO/数据库凭证硬编码**: 使用 `.env` 文件 + `.gitignore`

### P1（核心体验，1周内修复）
1. **RAG 未集成 LLM**: 添加 OpenAI Chat API 生成自然语言回答
2. **题目无解析展示**: 答题后显示 `explanation` 字段
3. **引用来源不可点击**: `source_url` 改为超链接
4. **无答题历史**: 新增"我的答题记录"页面

### P2（优化体验，2周内）
1. **Dashboard 图表移动端适配**: Recharts 响应式配置
2. **题库下拉改分页**: 支持随机推荐/分类筛选
3. **文档上传异步化**: 引入任务队列（Celery）
4. **增加缓存层**: Redis 缓存 KPI、热点检索

### P3（长期优化）
1. **微服务拆分**: API 与数据库解耦，支持水平扩展
2. **监控告警**: 集成 Prometheus + Grafana
3. **自动化测试**: CI/CD 流程中集成单元测试/E2E测试
4. **PWA 支持**: Service Worker 离线缓存

## 四、总结

### 产品层面
- **核心价值清晰**，但用户旅程需优化（明确主场景、减少摩擦）
- **RAG 体验不完整**（无 LLM 生成），影响产品差异化竞争力
- **题库功能基础薄弱**（无分类、无历史、防重复过严），难以支撑深度学习场景

### 技术层面
- **架构简洁但缺乏弹性**（单体架构、无缓存、无队列）
- **安全隐患严重**（CORS、凭证泄露、无 HTTPS）
- **代码质量中等**（类型注解好，但缺测试、缺类型检查、缺监控）

### 建议发展路径
1. **MVP验证期**（当前）: 修复 P0/P1 问题，确保核心流程可用
2. **产品打磨期**（1-3个月）: 完善用户体验（P2问题），增加数据分析
3. **规模化期**（3-6个月）: 架构重构（微服务、缓存、队列），支撑 10万+ 用户
