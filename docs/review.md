# 项目功能与实现问题审视

## 产品经理视角（功能层）
- 图片上传“自动切片向量化”体验与宣传不符：非文本文件不会做 OCR，后端仅用描述或标题入库，图片内容本身无法被检索（backend/app/documents.py:42-90）。应强制图片填写文字描述或补充 OCR/多模态嵌入。
- RAG 回答只是拼接命中的切片前 200 字，没有重写、去重或置信度过滤（backend/app/rag.py:12-36），用户看到的往往是片段堆叠而非“低幻觉回答”。需要增加生成式回答或最小相似度阈值。
- 互动题库缺乏学习闭环：后台题目含 `explanation` 字段，前台答题页未展示正确答案或解析（frontend/src/pages/Quiz.tsx:47-150），用户只能知道对错，无法获得知识补充。
- 管理端批量上传失败缺少回溯/重试：当前只标记“失败”且继续下一个任务（frontend/src/pages/Upload.tsx:51-105），未提示哪些文件已入库，容易重复上传或形成脏数据。需提供失败列表、重试或整体回滚。

## 全栈工程师视角（实现层）
- 文档入库缺乏一致性保障：`/api/documents/upload` 先上传到 MinIO 并提交 Document，再逐个嵌入切片；若 OpenAI/数据库写入失败，会留下对象与 Document 但无 Chunk，且异常未捕获（backend/app/documents.py:42-93）。应使用事务并在失败时清理已上传对象。
- 速率限制仅为单进程内存计数，既不持久也不跨实例，也不是漏桶/滑窗算法（backend/app/deps.py:17-32），与“鉴权与风控”承诺不符；在多 worker 或水平扩展下形同虚设。
- 嵌入调用缺少降级保护：OpenAI 请求失败会直接抛错返回 500，既不回退到哈希向量，也不向上层返回可控错误（backend/app/utils/embedding.py:16-27），会中断上传/问答链路。需捕获 httpx 异常并优先走本地 deterministic 向量。
